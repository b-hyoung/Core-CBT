PROMPT PATTERN · ⑤

재현 → 가설 → 최소 변화 검증

PROMPT PATTERN · ⑥

리뷰 · 리팩터링

변경 의도 vs 실제 diff 갭 확인

왜 이렇게

  • 스펙과 diff 가 벌어지는 게 가장 흔한 버그
  • 독립 검증자(서브 에이전트) 가 사각지대를 봄
  • PASS/WARN/FAIL 강제로 모호함 제거
ultrareview · code-reviewer 서브에이전트로 독립 검증
이 PR / 브랜치를 비판적으로 봐줘. 컨텍스트는 알려주지 말고 코드만 봐. 확인 포인트: 1. 스펙의 결정 테이블과 코드가 일치하는가 2. 무관한 변경이 섞였는가 3. 테스트가 행위 아닌 구현을 검증하지 않는가 4. 다음 사람이 5분 안에 흐름을 따라갈 수 있는가 5. 에러 처리가 시스템 경계(외부)에만 있는가 각 항목 PASS / WARN / FAIL + 한 줄 근거. 가장 위험한 1개 골라서 fix 제안.

교훈 5줄

  1. 스펙을 먼저 결정 테이블로 쓴다.
  2. 큰 결정은 한 번에 하나씩 객관식으로 묻는다.
  3. 에이전트는 프레임워크 없이 직접 짠다.
  4. AI 출력은 외부 검증기를 통과시킨다.
  5. 운영 데이터가 다음 기능 우선순위를 정한다.
